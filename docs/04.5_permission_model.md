# Purpose

This document defines the authorization model of WorkLedger.

It specifies:

* User roles
* Permissions
* Allowed actions
* Ownership rules
* Administrative capabilities

The objective is to ensure predictable, secure, and scalable access control while keeping the permission system simple.

---

# Philosophy

Permissions protect the integrity of the Ledger.

History should never be modified accidentally.

Every user should only access the information necessary to perform their responsibilities.

---

# Authorization Model

WorkLedger follows **Role-Based Access Control (RBAC)**.

Permissions are assigned to Roles.

Users receive permissions through their assigned Role.

The system authorizes actions based on permissions, not user identity.

---

# Core Roles

## Workspace Owner

The Workspace Owner has unrestricted access.

Responsibilities include:

* Create Workspace
* Manage members
* Configure settings
* Manage permissions
* Archive Workspace
* Restore archived records
* Administrative corrections

Typically the Founder, Manager, Team Lead, or Project Owner.

---

## Reviewer

A Reviewer manages work and evaluates contributors.

Responsibilities include:

* Create Assignments
* Assign Contributors
* Review Submissions
* Publish Reviews
* Request Revisions
* Complete Assignments
* View Workspace Analytics

A Reviewer cannot modify Workspace administration.

---

## Contributor

A Contributor performs assigned work.

Responsibilities include:

* View assigned work
* Accept Assignments
* Update progress
* Submit work
* Respond to revision requests
* View personal metrics
* View personal Ledger

Contributors cannot review themselves.

---

## Observer (Future)

Read-only access.

Suitable for:

* Clients
* Faculty
* Auditors
* Stakeholders

Observers cannot modify data.

# Ownership Rules

Ownership determines **what** may be edited.

Permissions determine **who** may edit.

Examples:

* A Contributor owns their Submission.
* A Reviewer owns the Review they publish.
* The Workspace owns all historical records.
* Metrics are owned by the System.

Ownership never overrides permission.

---

# Assignment Permissions

## Before Acceptance

Reviewer may:

* Edit Assignment
* Change deadline
* Modify description
* Reassign Contributor

Contributor may:

* View Assignment
* Accept Assignment

---

## After Acceptance

Reviewer may:

* Clarify requirements
* Extend deadline
* Cancel Assignment (administrative)

Contributor may:

* Update progress
* Upload work

Core Assignment details should remain stable.

---

## During Review

Contributor may:

* View review status

Contributor may **not**:

* Replace the active Submission
* Modify the Submission under review

Reviewer may:

* Publish Review
* Request Revision
* Approve Submission

---

## After Completion

Assignment becomes read-only.

Only Workspace Owners may perform administrative corrections.

Historical integrity takes precedence over convenience.

---

# Review Permissions

Only the assigned Reviewer may:

* Publish Review
* Edit an unpublished Review
* Request Revision

Published Reviews are immutable.

Corrections require Workspace Owner privileges and should generate an Activity.

---

# Submission Permissions

Contributors may:

* Create Submission
* Upload attachments
* Add notes

Contributors may not modify a Submission once it enters the review process.

Each revision creates a new Submission record.

---

# Administrative Corrections

The Workspace Owner may:

* Correct accidental data entry
* Restore archived records
* Resolve permission conflicts

Administrative corrections should always generate an Activity with:

* Timestamp
* User
* Reason
* Previous value
* Updated value

The Ledger must preserve transparency.

---

# Security Principles

1. Deny by default.
2. Grant only the minimum required permissions.
3. Preserve historical integrity.
4. Log all administrative actions.
5. Never silently modify historical records.

---

# Future Permission Extensions

The permission model supports future roles such as:

* Team Lead
* Mentor
* Department Head
* Client
* External Reviewer
* HR Manager
* Organization Administrator

New roles should reuse existing permissions wherever possible.

---

# Guiding Principle

Roles describe **responsibilities**.

Permissions authorize **actions**.

Ownership determines **scope**.

Together, they protect the integrity of the WorkLedger while enabling efficient collaboration.
