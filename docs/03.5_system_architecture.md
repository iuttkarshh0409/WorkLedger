# Purpose

This document defines the high-level architecture of WorkLedger.

It describes how information flows through the system, how responsibilities are separated, and how every layer interacts.

This architecture is intentionally independent of any specific frontend framework, backend technology, or database.

---

# Architectural Philosophy

WorkLedger follows a **Data-Driven Architecture**.

Every piece of information displayed in the application originates from structured data.

No visual component owns business logic.

No analytics are manually maintained.

No dashboard stores its own state beyond presentation.

The application exists to transform structured records into meaningful interfaces.

---

# Core Architectural Principle

```
User Action
      │
      ▼
Structured Data
      │
      ▼
Business Logic
      │
      ▼
Derived Intelligence
      │
      ▼
Presentation Layer
```

Every feature follows this pipeline.

---

# System Layers

The application is divided into five independent layers.

```
Presentation Layer

↓

Application Layer

↓

Business Layer

↓

Data Layer

↓

Infrastructure Layer
```

Each layer has a single responsibility.

---

# Presentation Layer

## Responsibility

Display information.

Capture user interaction.

Render visual components.

The Presentation Layer should never perform calculations.

It simply requests processed data and renders it.

Examples include:

* Dashboard
* Contributor Profile
* Assignment Board
* Timeline
* Review Form
* Analytics Pages
* Workspace Settings

---

# Application Layer

## Responsibility

Coordinate user workflows.

This layer manages interactions between the interface and business services.

Examples:

* Creating Assignments
* Publishing Reviews
* Completing Assignments
* Uploading Submissions
* Switching Workspaces

The Application Layer orchestrates actions.

It does not calculate business rules.

---

# Business Layer

## Responsibility

This is the heart of WorkLedger.

It contains all business logic.

Examples:

* Score Calculation
* Deadline Validation
* Review Rules
* Permission Rules
* Contributor Metrics
* Workspace Analytics
* Milestone Progress
* Timeline Generation

If the platform's behavior changes, this is the layer that changes.

---

# Data Layer

## Responsibility

Store objective facts.

The Data Layer knows nothing about dashboards or UI.

It stores:

* Contributors
* Assignments
* Reviews
* Activities
* Milestones
* Submissions

Nothing else.

---

# Infrastructure Layer

## Responsibility

Provide technical capabilities.

Examples:

* Local JSON Storage
* Database
* Authentication
* Cloud Storage
* GitHub API
* Google Drive API
* Notification Services
* AI Providers

Infrastructure can change without affecting business logic.

---

# Information Flow

Every feature follows the same lifecycle.

```
User

↓

Application

↓

Business Rules

↓

Data Storage

↓

Derived Services

↓

UI Update
```

This creates predictable behavior throughout the platform.

---

# Derived Services

One of WorkLedger's defining architectural concepts is the Derived Service.

A Derived Service calculates information from existing records.

It never stores permanent data.

Examples include:

## Contributor Metrics Service

Calculates:

* Average Score
* Completion Rate
* Contribution Streak
* Revision Count
* Skill Distribution

---

## Workspace Analytics Service

Calculates:

* Active Contributors
* Average Workspace Performance
* Assignment Distribution
* Completion Trends

---

## Timeline Service

Generates chronological event history.

No timeline is stored directly.

It is reconstructed from Activities.

---

## Milestone Progress Service

Calculates:

* Progress Percentage
* Remaining Work
* Pending Reviews
* Completion Forecast

---

## AI Insight Service

Generates:

* Performance Summaries
* Growth Reports
* Review Suggestions
* Contributor Highlights

This service enhances existing information.

It never becomes the primary source of truth.

---

# Source of Truth

Only the Data Layer owns persistent information.

Every other layer consumes or transforms data.

```
Stored Facts

↓

Derived Facts

↓

Visual Representation
```

If information can be regenerated, it should not be stored.

---

# Event-Driven Thinking

Every meaningful action produces an Activity.

Examples:

Assignment Created

↓

Assignment Accepted

↓

Submission Uploaded

↓

Review Published

↓

Assignment Completed

The Activity stream becomes the foundation for:

* Timelines
* Notifications
* Analytics
* Audit History
* AI Summaries

---

# Component Independence

Each module should remain independently replaceable.

Examples:

Changing authentication should not affect assignments.

Changing storage should not affect analytics.

Changing AI providers should not affect reviews.

Changing frontend frameworks should not affect business rules.

Loose coupling is a primary architectural goal.

---

# Architectural Constraints

The following practices are prohibited:

* Business logic inside UI components
* Duplicate sources of truth
* Manually maintained analytics
* Circular dependencies between modules
* Framework-specific business logic
* Hardcoded contributor metrics

Violations should be treated as architectural defects.

---

# Engineering Principle

The architecture should remain understandable to a new contributor within a single reading session.

Complexity should emerge from solving real problems, not from unnecessary abstraction.

Every module should have one clear responsibility.

Every layer should communicate in one direction.

Every decision should reinforce the principles established in the WorkLedger documentation.

---

# Architecture Motto

> **Store facts. Derive intelligence. Present clarity.**
