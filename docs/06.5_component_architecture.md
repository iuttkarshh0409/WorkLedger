# Purpose

This document defines the component hierarchy of WorkLedger.

Its purpose is to establish a reusable, modular, and scalable UI architecture before implementation begins.

Every screen within the application should be composed from reusable building blocks rather than page-specific implementations.

---

# Component Philosophy

Components represent reusable interface behavior.

Pages assemble components.

Components should never be tightly coupled to individual pages.

The same component should be usable throughout the application whenever appropriate.

---

# Component Hierarchy

```text
Application

└── Workspace

    ├── Layout
    │
    ├── Navigation
    │
    ├── Feature Modules
    │
    └── Shared Components
```

---

# Layer 1

## Application Shell

The Application Shell provides the persistent application structure.

Responsibilities:

* Sidebar
* Header
* Breadcrumb
* Global Search
* Notification Center
* Theme Provider
* Workspace Switcher (Future)

This layer never changes between pages.

---

# Layer 2

## Workspace Layout

Every Workspace shares a common layout.

```text
Workspace

├── Sidebar

├── Top Navigation

├── Main Content

└── Context Panel (Future)
```

Only the Main Content changes during navigation.

---

# Layer 3

## Feature Modules

Feature Modules own business-specific interfaces.

Modules include:

* Dashboard
* Contributors
* Milestones
* Assignments
* Reviews
* Activity
* Analytics
* Settings

Each module contains independent reusable components.

---

# Shared Components

These components should never contain business logic.

---

## Cards

Reusable information containers.

Examples:

* Contributor Card
* Assignment Card
* Milestone Card
* Review Card
* Metric Card
* Activity Card

Every card follows a consistent visual structure.

---

## Tables

Reusable data presentation.

Examples:

* Contributors Table
* Assignment Table
* Reviews Table
* Activity Table

Tables should support:

* Sorting
* Filtering
* Pagination
* Search

---

## Status Badge

Displays standardized status values.

Examples:

* Assigned
* In Progress
* Submitted
* Completed
* Archived

Status colors and labels should be centralized.

---

## Priority Badge

Displays:

* Low
* Medium
* High
* Critical

Visual consistency is mandatory.

---

## Progress Components

Includes:

* Progress Bar
* Circular Progress
* Completion Indicator

These visualize derived metrics.

---

## Timeline

Displays chronological Activities.

Used in:

* Contributor Profile
* Assignment Detail
* Workspace Activity

Timeline consumes Activity data only.

---

## Metric Card

Displays one derived metric.

Examples:

* Completion Rate
* Active Assignments
* Pending Reviews
* Average Score

Metric Cards never calculate values.

---

## Score Indicator

Displays category scores.

Used throughout:

* Reviews
* Contributor Profile
* Analytics

Should support:

* Numeric score
* Color indication
* Tooltip description

---

## Contributor DNA Card

A signature WorkLedger component.

Displays the contributor's long-term competency profile.

Dimensions include:

* Technical Quality
* Documentation
* Communication
* Ownership
* Problem Solving
* Timeliness

The DNA Card is generated entirely from historical Review data.

It is never manually edited.

Future versions may support:

* Radar Chart
* Skill Heatmap
* Trend Overlay
* AI-generated narrative

---

## Activity Feed

Displays recent Activities.

Supports:

* Filtering
* Infinite scrolling (Future)
* Relative timestamps

---

## Empty State

Every module should provide meaningful empty-state components.

Examples:

* No Assignments
* No Contributors
* No Reviews

Empty States should encourage the next action.

---

## Dialog Components

Reusable dialogs include:

* Create Assignment
* Review Submission
* Archive Assignment
* Delete Confirmation
* Invite Contributor

Dialogs should remain stateless whenever possible.

---

## Form Components

Reusable form elements include:

* Text Input
* Rich Text Editor
* Select Dropdown
* Date Picker
* Priority Selector
* Status Selector
* Tag Selector
* Score Slider
* File Upload

Business validation belongs outside these components.

---

# Component Communication

Components communicate through clearly defined interfaces.

```text
Data

↓

Container Component

↓

Presentation Component

↓

User Interaction

↓

Application Layer
```

Presentation components should never modify business state directly.

---

# Smart vs Presentational Components

## Smart Components

Responsible for:

* Data retrieval
* Business interaction
* State orchestration

Examples:

* Contributor Profile Container
* Assignment Container
* Dashboard Container

---

## Presentational Components

Responsible only for rendering.

Examples:

* Metric Card
* Timeline Item
* Status Badge
* Assignment Card

Presentational components receive data through properties.

---

# Component Naming Convention

All reusable components should use consistent naming.

Examples:

```text
ContributorCard

AssignmentCard

ReviewCard

MetricCard

StatusBadge

TimelineItem

ScoreIndicator

ContributorDNA

ActivityFeed

ProgressRing

EmptyState
```

Names should describe responsibility rather than appearance.

---

# Reusability Principles

Before creating a new component, ask:

1. Does a similar component already exist?
2. Can an existing component be extended?
3. Does this belong to a Feature Module or Shared Components?
4. Will another module benefit from this component?

If yes, reuse before creating.

---

# Performance Principles

Components should:

* Render only required data.
* Avoid unnecessary re-renders.
* Remain independent.
* Support lazy loading where appropriate.

Optimization follows correctness.

---

# Accessibility Principles

Every component should support:

* Keyboard navigation
* Screen readers
* Semantic HTML
* Visible focus states
* Sufficient color contrast

Accessibility is a baseline requirement.

---

# Guiding Principle

Components are the vocabulary of the interface.

Small, reusable, and predictable components enable WorkLedger to scale without increasing complexity.

Every page should feel assembled, never handcrafted.
